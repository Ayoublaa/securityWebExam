Chapitre 2 — Injection SQL
SQLDéfinition & principe de l'injection SQL
▼
Injection SQL = données utilisateur non protégées + requête SQL construite directement = vulnérabilité.

Comment ça marche :

  • L'application insère les données saisies directement dans une requête SQL sans validation.
  • L'attaquant modifie la logique de la requête en injectant du code SQL.
  • Vecteurs : formulaires, champs login/mot de passe, URL, paramètres GET/POST.

Code vulnérable typique :

$sql = "SELECT * FROM users WHERE login = '" . $_POST['login'] . "'"; // Attaquant saisit : ' OR '1'='1 // Résultat : WHERE login = '' OR '1'='1' → TOUJOURS VRAI

Impacts (CIA) :

Confidentialité → vol de données Intégrité → modification/suppression Disponibilité → DROP TABLE...
SQLTypes d'injection SQL & protections
▼

Types d'injection :

TypeDescription
Error-basedExploite les messages d'erreur SQL pour extraire des infos
Union-basedUtilise UNION SELECT pour combiner des résultats de tables
Blind SQLL'attaquant ne voit pas directement le résultat (inférence)
Time-basedUtilise le délai de réponse (SLEEP()) pour inférer des données

Protection principale — PDO + requêtes préparées :

// Avec marqueurs ? (placeholders) $req = $pdo->prepare("SELECT * FROM users WHERE login = ? AND pass = ?"); $req->execute([$login, $pass]); // Avec paramètres nommés $req = $pdo->prepare("SELECT * FROM users WHERE login = :login AND pass = :pass"); $req->execute(['login' => $login, 'pass' => $pass]);
Les ? / :nom séparent le code SQL des données → l'injection devient impossible.

Aussi : filtrer les entrées (strlen, preg_match), ne jamais faire confiance aux données utilisateur, magic_quotes_gpc est obsolète depuis PHP 8.0.

Chapitre 3 — XSS (Cross-Site Scripting)
XSSDéfinition, types et différence avec SQL Injection
▼
XSS = injection JavaScript côté client. Cible = navigateur de la victime (≠ SQL Injection qui cible le serveur).

3 types de XSS :

TypeStocké ?DangerMécanisme
ReflectedNonMoyenScript renvoyé immédiatement dans la réponse HTTP (via lien piégé)
StoredOui ✓Élevé ⚠️Script enregistré en BDD, touche tous les visiteurs
DOM-BasedNonVariableJavaScript côté client qui manipule mal des données de l'URL

Effets possibles :

  • Redirection vers un site malveillant
  • Affichage de messages indésirables
  • Vol de cookies (dans les vieilles apps)
  • Exécution d'actions à la place de l'utilisateur
XSSProtections contre XSS — CSP & Nonce
▼

A. Filtrer les entrées utilisateur

Toute donnée venant de l'utilisateur est non fiable. Vérifier type, longueur, format, caractères spéciaux.

B. Échapper les sorties HTML

// Transformer <script> en texte inoffensif echo htmlspecialchars($input, ENT_QUOTES, 'UTF-8'); // ou echo htmlentities($input, ENT_QUOTES, 'UTF-8');

C. Supprimer les balises

$propre = strip_tags($_POST['message']);

D. Content Security Policy (CSP)

CSP = directive HTTP qui dit au navigateur quels scripts sont autorisés. Agit comme un pare-feu côté navigateur.

E. Nonce CSP

$nonce = base64_encode(random_bytes(16)); // Dans le header HTTP : Content-Security-Policy: script-src 'nonce-{$nonce}' // Dans le HTML : <script nonce="{$nonce}">...</script>

Le nonce doit être aléatoire, unique par requête. Utiliser random_bytes() et non rand() ou mt_rand().

Chapitre 4 — CSRF (Cross-Site Request Forgery)
CSRFMécanisme & exemple d'attaque CSRF
▼
CSRF = forcer un utilisateur authentifié à exécuter une action non voulue. Le pirate exploite la session active, pas le mot de passe.

Schéma de l'attaque :

  • 1. L'admin est connecté à son back-office (session + cookies actifs)
  • 2. Le pirate lui envoie un email/message contenant une image piégée
  • 3. L'admin ouvre le message → le navigateur charge l'image
  • 4. En réalité = appel HTTP vers admin/cours/supp.php?id=120
  • 5. Le serveur reçoit la requête avec les cookies de l'admin → action exécutée
<!-- Image piégée dans un email --> <img src="https://site.com/admin/cours/supp.php?num=120" />

Actions possibles :

Suppression d'un enregistrement Modification d'un mot de passe Ajout d'un compte admin Exécution d'une action dangereuse
CSRFProtections contre CSRF
▼

1. Ne jamais faire une action sensible avec GET

Un lien GET peut être déclenché automatiquement par une image ou un lien. Toujours utiliser POST pour les modifications.

2. Token CSRF ← protection principale ⭐

// Génération côté serveur $_SESSION['csrf'] = bin2hex(random_bytes(32)); <!-- Dans le formulaire HTML --> <input type="hidden" name="csrf_token" value="<?= $_SESSION['csrf'] ?>"> // Vérification à la réception if($_POST['csrf_token'] !== $_SESSION['csrf']) { die('CSRF détecté'); }
Le token CSRF prouve que la requête vient d'un formulaire légitime du site. Le pirate ne peut pas connaître cette valeur.

3. HTTP_REFERER (complémentaire, insuffisant seul)

Vérifie la page d'origine mais peut être absent ou manipulé. Ne jamais s'y fier seul.

4. Demander confirmation pour les actions irréversibles

Afficher "Voulez-vous vraiment supprimer ?" — utile mais doit être combiné avec un token CSRF.

Chapitre 5 — RCE / Injection de code
RCERemote Code Execution — définition & protections
▼
RCE = Remote Code Execution. Le serveur exécute directement une entrée utilisateur comme du code PHP/système.

Exemple dangereux :

// Code vulnérable — JAMAIS faire ça $code = $_GET['code']; eval($code . ";"); // Attaquant envoie : ?code=phpinfo() ou ?code=system('rm -rf /')

Impacts (CIA — les 3) :

PilierConséquence
ConfidentialitéLire le code source, la config, les identifiants BDD
IntégritéModifier la BDD, changer le contenu du site
DisponibilitéSupprimer des fichiers, bloquer le serveur

Protections :

  • Éviter eval() — ne jamais exécuter une entrée utilisateur
  • Valider les entrées — type, longueur, format, caractères autorisés
  • Liste blanche — n'autoriser que les actions explicitement prévues
  • Ne jamais passer du code dans l'URL
SQL Injection → injecte du SQL. RCE → injecte du code PHP ou des commandes système. Même cause racine : confiance aveugle dans les entrées.

Règle d'or pour le stockage des mots de passe :

// Ne jamais stocker en clair ! $hash = password_hash($password, PASSWORD_BCRYPT); // Vérification if(password_verify($password, $hash)) { /* connecté */ }
HTTP = données interceptables. HTTPS = chiffrement TLS. Toujours HTTPS sur la page de login !
